Fast Start

Pick the easiest path for your host.

1. Shared-host zip

Get hubroute-php-sqlite.zip, upload it in cPanel/File Manager, extract it into public_html, and open /health.php.

2. One-command SSH

Extract the zip locally, then sync the extracted runtime folder with rsync.

3. LLM-assisted deploy

Point an LLM at the repo and tell it to deploy only the PHP + SQLite app, keeping SQLite as production storage.

4. Vercel production

Use the separate Vercel-ready fileset with hosted SQLite-compatible storage. The maintainer demo is not a user deployment path.

Deployment Targets

Choose the host before choosing the command.

Production

Shared hosting / cPanel

Use the PHP + SQLite zip. No terminal is required if you already have the zip: upload it, extract it, open /health.php, then schedule php maintenance.php run --apply.

Production

VPS / SSH host

Extract the zip locally and sync the runtime folder to Apache, Nginx, or PHP-FPM. Keep the SQLite data directory writable and private.

rsync -az --delete hubroute-php-sqlite/ USER@HOST:/home/USER/public_html/
Starter Artifact

Vercel production starter

Use dist/hubroute-vercel.zip from vercel-app/: extract it, put the contents in GitHub, import the repo in Vercel, add Turso/libSQL env vars, and deploy. The static demo is maintainer-only.

Why PHP + SQLite

Production starts where operators already host software.

HubRoute uses PHP + SQLite deliberately. Many courier, merchant, and local-operations teams in emerging markets begin with shared hosting, cPanel, FTP/SFTP, or a small VPS instead of containers, managed databases, and always-on app servers.

Compatibility

Runs on older infrastructure

PHP 8 + SQLite works on common LAMP-style hosts, shared hosting plans, Apache/Nginx/PHP-FPM setups, and small VPSes.

Cost

Low startup cost

No managed database, queue, Redis service, object storage account, container registry, or always-on app server is required.

Operations

Simple backup and support

SQLite keeps early production data in one file, so backup, restore, staging copies, and support handoff stay straightforward.

Runtime

Host-managed workers

PHP's request-scoped lifecycle lets the host manage workers, memory cleanup, process restarts, HTTPS, and logs.

Security

Mature platform primitives

The app uses PDO prepared statements, password_hash(), standard sessions, CSRF tokens, CSP headers, and role checks.

Upgrade Path

Start simple, move later

The same product rules can move to a Vercel-native or managed-database version later without raising the first deployment cost.

Zip Deploy

Shared hosting can be a zip extraction.

1. Upload

Upload hubroute-php-sqlite.zip to public_html or the subdomain folder.

2. Extract

Extract the zip and make sure index.php, hubroute.php, and health.php are directly in the website folder.

3. Open

Visit /health.php, then /. First load creates SQLite.

  • Log in to cPanel or your host panel and open File Manager.
  • Turn on hidden files if your panel has that option; .htaccess and .env.example are supposed to be visible after extraction.
  • Open public_html or the folder for your subdomain.
  • Upload hubroute-php-sqlite.zip.
  • Click Extract.
  • If extraction creates a hubroute-php-sqlite folder, move the files inside it up into public_html unless you want the app at that folder path.
  • Delete the zip after extraction.
  • Set PHP to 8.1+ and enable pdo_sqlite plus sqlite3.
  • Open https://YOUR-DOMAIN/health.php, then https://YOUR-DOMAIN/.

The zip contains the PHP app, CLI maintenance command, health check, environment example, root denial rules, empty protected data/ folder, and a tiny README.txt.

SSH

Sync an extracted zip to a VPS.

Local smoke run

Use PHP's built-in server to confirm the app boots before uploading.

php -S 127.0.0.1:8080

SSH deploy

Replace USER, HOST, and the host path.

rsync -az --delete hubroute-php-sqlite/ USER@HOST:/home/USER/public_html/

Use the built-in PHP server for local smoke tests only. For production traffic, run through your shared host, Apache, Nginx, or PHP-FPM setup.

Shared Hosting

Deploy the PHP + SQLite backend.

  • Set the domain or subdomain to PHP 8.1+ and enable pdo_sqlite plus sqlite3.
  • Upload and extract hubroute-php-sqlite.zip into the app directory.
  • Make sure hubroute.php, maintenance.php, health.php, index.php, .htaccess, .env.example, and data/ are directly in that app directory.
  • If .htaccess or .env.example is missing, enable hidden files in the host file manager and check again.
  • Create a writable data/ directory, or copy .env.example to .env and set DATA_DIR to a private path outside public_html.
  • Leave DB_PATH commented unless the SQLite file needs a different path from DATA_DIR/hubroute.sqlite.
  • Visit /health.php and confirm PHP version, PDO SQLite, SQLite3, directory writability, database writability, and SQLite write all pass.
  • Visit the app domain root or /hubroute.php?r=login&demo=hub to create the database and seed accounts.
  • Confirm /data/hubroute.sqlite is not downloadable. If it is reachable, move DATA_DIR outside the web root immediately.
First Run

Lock down the SQLite app before real operations.

1. Rotate seeded passwords

Use Admin -> Users to create a production admin, then reset or disable all seeded admin, hub, rider, and customer users.

2. Protect runtime files

Keep .env, data/hubroute.sqlite, logs, and backups private. Use the strictest permissions your host supports.

3. Back up SQLite

Schedule backups for data/hubroute.sqlite and test a restore before live courier use.

4. Review audit logs

Admin users can inspect privileged and custody-affecting changes under Admin -> Audit.

Credential Rotation

Replace seed logins before production data enters the system.

  • Log in as admin@hubroute.local with admin1234.
  • Open Admin -> Users and create a production admin with a real email address and a unique temporary password.
  • Log out, then log back in as the production admin.
  • For each seeded user, enter a new unique password or set status to disabled.
  • Disable admin@hubroute.local only after the production admin account is confirmed.
  • Confirm old seed passwords no longer work for admin, hub, agent, and customer accounts.
  • Open Admin -> Audit and confirm the user access changes were recorded.
  • Run php maintenance.php backup after the credential rotation is complete.

Do not enter real parcel, merchant, rider, or customer data until seeded credentials have been rotated or disabled.

How It Works

Custody is explicit from booking to delivery attempt.

Intake

Create parcels with merchant, address, payment, weight, and service notes; the system assigns the tracking number and routes each parcel into a pickup hub queue.

Sort and dispatch

Receive incoming parcels, clear old local assignment, sort by destination, and send onward through line-haul or last-mile runs.

Track and reconcile

Record events, delivery attempts, COD collection, and public tracking updates without exposing full PII.

Vercel

Use the separate Vercel fileset, not the demo.

Not the PHP zip

Do not upload hubroute-php-sqlite.zip to Vercel. The zip is for PHP hosts. Vercel production needs a separate app built for Vercel-supported server runtimes.

Vercel-ready fileset

HubRoute now ships a second artifact: dist/hubroute-vercel.zip, generated from vercel-app/. It contains Vercel app source, hosted SQLite setup, health, setup, tracking, cron, and env example files.

Keep SQLite semantics

Do not use local filesystem SQLite on Vercel production. Use hosted SQLite-compatible storage such as Turso/libSQL if the product must remain SQLite-based.

Extend the backend

The starter includes public tracking and maintenance plumbing. Full auth, RBAC, parcel mutations, CSV export, and admin user access control can be layered into this Vercel artifact next.

Do not deploy the static browser walkthrough as a user deployment. It is maintainer-only. Also do not deploy the PHP + local SQLite zip as a Vercel production backend; the zip is for PHP hosts.

Route mapping

health.php is /api/health. Public tracking is /track/[code] backed by /api/track?code=.... Setup is /setup backed by protected /api/setup.

Maintenance

maintenance.php run --apply becomes a protected /api/cron/maintenance route called by Vercel Cron. The route must verify CRON_SECRET.

Environment

Configure TURSO_DATABASE_URL, TURSO_AUTH_TOKEN, SESSION_SECRET, CRON_SECRET, SETUP_SECRET, APP_TIMEZONE, and retention settings.

Deploy flow

Extract hubroute-vercel.zip, upload the contents to GitHub, import the repo in Vercel, add env vars, open /api/health, run /setup, then test /track/HR260703DHK1A2.

Still required

The fileset cannot include real production secrets, a linked Vercel project, a provisioned hosted database, or the deployer's custom domain. Those are configured during deployment.

Beginner dashboard steps

This path uses GitHub and the Vercel dashboard. The deployer does not need local npm for these steps.

  • Get dist/hubroute-vercel.zip.
  • Extract the zip on your computer.
  • Open the extracted folder and confirm api/, lib/, scripts/, index.html, track.html, setup.html, styles.css, package.json, package-lock.json, vercel.json, .env.example, and README.md are at the top level.
  • Create a new empty GitHub repository, for example hubroute-vercel.
  • Upload the contents of the extracted folder into the root of that repository. Do not upload the parent folder itself.
  • Do not upload the PHP shared-host zip, browser walkthrough files, .env, local SQLite databases, logs, or backup files.
  • Create a hosted SQLite-compatible database in Turso/libSQL.
  • Copy the database URL into TURSO_DATABASE_URL.
  • Create a database token and copy it into TURSO_AUTH_TOKEN.
  • Create unique 32+ character secrets for SESSION_SECRET, CRON_SECRET, and SETUP_SECRET.
  • In Vercel, choose Add New -> Project.
  • Import the GitHub repository created above.
  • Keep the root directory as the repository root. If the files are inside a subfolder, choose the folder that contains vercel.json.
  • If Vercel asks for a framework, choose Other or No Framework.
  • Keep advanced command fields at their defaults.
  • Add Production env vars: TURSO_DATABASE_URL, TURSO_AUTH_TOKEN, SESSION_SECRET, CRON_SECRET, SETUP_SECRET, APP_TIMEZONE, and the maintenance retention settings.
  • Click Deploy.
  • Open https://YOUR-VERCEL-DOMAIN/api/health.
  • If health reports missing env vars, add the missing values in Vercel and redeploy.
  • Open https://YOUR-VERCEL-DOMAIN/setup and enter SETUP_SECRET.
  • Open https://YOUR-VERCEL-DOMAIN/track/HR260703DHK1A2.
  • In Vercel project settings, confirm the scheduled cron points at /api/cron/maintenance.

Security

Seeded credentials and production credentials are separate concerns.

  • Do not reuse seeded emails or passwords in production.
  • Use Admin -> Users to create production admins, reset passwords, and disable unused or seeded accounts.
  • Keep environment files and SQLite data outside public web paths when possible.
  • Use HTTPS, secure session settings, CSRF protection, prepared SQL, and escaped output for the PHP app.
  • Use role-based checks for parcel, route, rider, hub, and export operations.
Operations

Automate backups and old-data cleanup.

  • Run php maintenance.php prune first to preview what would be removed.
  • Run php maintenance.php run --apply from cron to back up SQLite, prune old rows, and vacuum after cleanup.
  • Terminal parcels in delivered, failed, or returned are removed only after the configured retention window; active parcels are kept.
  • Backups default to data/backups and are retained for 30 days unless MAINTENANCE_BACKUP_KEEP_DAYS changes.
  • Test restore by copying a backup into a separate staging folder and opening /health.php.
  • Review data/php-error.log after deployments and keep ENABLE_EXTRA_LOGGING=false outside troubleshooting windows.
Troubleshooting

Common setup and deployment checks.

  • If /health.php fails, enable PHP 8.1+, pdo_sqlite, and sqlite3, then confirm the data directory is writable.
  • If first login works but actions fail, check data/php-error.log and host write permissions.
  • If /data/hubroute.sqlite downloads in a browser, move DATA_DIR outside the web root immediately.
  • If the PHP app fails on a host, confirm PHP 8.1+, PDO SQLite, writable data directory, and private environment configuration.
